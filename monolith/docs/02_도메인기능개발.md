# 기본 사항
- monolith 프로젝트에 주문, 서비스, 결제 도메인을 개발합니다.
- 각 도메인은 별도의 패키지로 구성합니다.
- 주문 서비스만 컨트롤러(API)를 제공합니다.
- 포인트 서비스와 결제 서비스는 주문 서비스에서 호출하는 방식으로 구현합니다.

# 데이터 흐름

1. 사용자가 상품 주문을 요청합니다.
   1. 서비스는 상품 정보를 관리하지 않기 때문에 주문 요청 시 상품 가격을 함께 전달받습니다.
2. 주문 서비스는 주문 정보를 확인합니다.
   1. 상품 가격, 포인트 사용 여부, 사용자 정보 등 
3. 주문 서비스는 포인트 서비스에 사용자 포인트 사용 요청을 보냅니다.
4. 포인트 서비스는 사용자 포인트를 차감하고 처리 결과를 주문 서비스에 반환합니다.
5. 포인트 서비스가 정상 응답을 보내면 주문 서비스는 결제 서비스에 결제 요청을 보냅니다.
   1. 결제 요청 금액 = 상품 가격 - 포인트 사용 금액
6. 결제 서비스는 외부 결제 API를 호출하여 결제를 처리하고 결과를 주문 서비스에 반환합니다.
   1. 실제로 외부 결제 API 서버는 없기 때문에 약간의 딜레이를 가지는 함수를 호출하여 결제 처리 시뮬레이션
7. 결제 서비스의 응답에 따라 주문 서비스는 주문 상태를 업데이트하고 최종 결과를 사용자에게 반환합니다.

## **주문 서비스:** 사용자 상품 주문(구매) 관리

- 사용자로부터 상품 주문 요청을 받아 주문을 생성하고 관리하는 서비스입니다.
- 주문 생성 시 포인트 서비스와 결제 서비스를 호출하여 주문 처리 로직을 수행합니다.
- 주문 상태를 관리합니다.

## **포인트 서비스(point):** 사용자 포인트 관리

- 사용자 포인트를 관리하는 서비스입니다.
- 주문 서비스에서 포인트 사용 요청을 받아 사용자 포인트를 차감하고 결과를 주문 서비스에 반환합니다.
- 사용자 정보가 없을 경우 새로운 사용자 정보를 생성하고 기본 포인트(10000)를 설정합니다.

## **결제 서비스(payment):** 외부 서버의 카드 결제 API 호출

- 외부 결제 API를 호출하여 결제를 처리하는 서비스입니다.
- 주문 서비스에서 결제 요청을 받아 외부 결제 API를 호출하여 결제를 처리하고 결과를 주문 서비스에 반환합니다.
- 실제로 외부 결제 API 서버는 없기 때문에 약간의 딜레이를 가지는 함수를 호출하여 결제 처리 시뮬레이션합니다.

# 테스트
- monolith 프로젝트 루트 디렉토리에 http 파일을 작성해서 API 테스트할 수 있도록 합니다.

# 추가 사항
- 각 도메인 로직을 의도적으로 실패하여 예외 상황을 직접 확인할 수 있도록 실패 코드를 작성하고 주석 처리합니다.
