# 기본 사항
- monolith 프로젝트에 다음 도메인을 개발합니다.
- 각 도메인은 별도의 패키지로 구성합니다.
- 주문 서비스만 컨트롤러(API)를 제공합니다.
- 포인트 서비스와 결제 서비스는 주문 서비스에서 호출하는 방식으로 구현합니다.

## **주문 서비스(order):** 사용자 주문(구매) 관리

- 테이블
    - **id:** 주문 고유 식별자 (PK)
    - **user_id:** 주문한 사용자 식별자
    - **product_id:** 상품 식별자
    - **total_amount:** 주문 총액 (포인트 + 결제금액)
    - **point_amount:** 사용할 포인트 (주문에 적용된 포인트)
    - **status:** 주문 상태 (PENDING, COMPLETED, CANCELLED)
    - **created_at:** 주문 생성 시간
    - **updated_at:** 주문 상태 수정 시각
- 기능
    - 주문 생성

## **포인트 서비스(point):** 사용자 포인트 관리

- 테이블
    - **id:** 포인트 거래 식별자 (PK)
    - **user_id:** 사용자 식별자 (FK)
    - **balance:** 현재 잔여 포인트
    - **created_at:** 최초 포인트 충전 시각
    - **updated_at:** 포인트 사용 시각
- 기능
    - 사용자 포인트 사용(사용자 정보 없을 경우 생성후 포인트 사용)

## **결제 서비스(payment):** 외부 서버의 카드 결제 API 호출

- 테이블
    - **id:** 결제 고유 식별자 (PK)
    - **order_id:** 주문 식별자 (FK)
    - **amount:** 결제 금액
    - **status:** 결제 상태 (READY, SUCCESS, FAIL)
    - **created_at:** 결제 시도 시간
    - **updated_at:** 결제 상태 수정 시각
- 기능
    - 외부 결제 API 호출

# 데이터 흐름

1. 사용자가 주문을 생성합니다.
2. 주문 서비스는 포인트 서비스에 사용자 포인트 사용 요청을 보냅니다.
3. 포인트 서비스는 사용자 포인트를 차감하고 결과를 주문 서비스에 반환합니다.
4. 주문 서비스는 결제 서비스에 결제 요청을 보냅니다.
5. 결제 서비스는 외부 결제 API를 호출하여 결제를 처리하고 결과를 주문 서비스에 반환합니다.
    1. 이때 외부 결제 서버는 실제로 존재하지 않기 때문에 API는 실제로 호출하지 않고 호출한 것과 같이 기능을 모킹합니다.
6. 주문 서비스는 주문 상태를 업데이트하고 최종 결과를 사용자에게 반환합니다.
